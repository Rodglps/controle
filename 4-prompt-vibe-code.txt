## CONTEXTO DO PROJETO

Você é um arquiteto de software sênior especialista em Java. Vamos construir juntos o sistema "Controle de Arquivos" — responsável pela entrada de dados para conciliação de cartão.

O sistema coleta arquivos EDI de adquirentes via SFTP e APIs, identifica o cliente e layout de cada arquivo, e os encaminha para um destino (S3 ou SFTP interno), mantendo rastreabilidade completa.

---

## STACK E INFRAESTRUTURA

- Java 17+ com Spring Boot 3
- Kubernetes (manifests YAML) + Docker Compose para ambiente local
- RabbitMQ para orquestração entre pods (servidor local via Docker para dev)
- Oracle Database (servidor local via Docker para dev)
- AWS S3 (LocalStack para dev local)
- SFTP (servidor local via Docker para dev)
- Mono repositório

---

## ARQUITETURA: 2 PODS PRINCIPAIS

### Pod 1 — Orquestrador
Responsabilidades:
- Conectar ao banco e carregar configurações de servidores SFTP (tabelas: server, sever_paths, sever_paths_in_out)
- Listar arquivos nas pastas SFTP configuradas
- Registrar cada arquivo encontrado na tabela file_origin (status inicial: COLETA / EM_ESPERA)
- Publicar mensagens no RabbitMQ com um ou mais arquivos por mensagem
- Controle de concorrência via tabela job_concurrency_control

### Pod 2 — Processador
Responsabilidades:
- Consumir mensagens do RabbitMQ
- Baixar o arquivo do SFTP usando streaming (sem carregar tudo em memória)
- Identificar o CLIENTE pelo nome do arquivo usando regras da tabela customer_identification_rule (critérios: COMECA-COM, TERMINA-COM, CONTEM, IGUAL) — todas as regras ativas devem ser satisfeitas
- Identificar o LAYOUT pelo nome do arquivo ou pelos primeiros 7000 bytes, usando regras da tabela layout_identification_rule (mesma lógica: todas as regras devem ser true)
- Fazer upload do arquivo (streaming) para o destino: S3 ou SFTP interno
- Atualizar file_origin, file_origin_client e file_origin_client_processing com os dados do processamento

## MODELAGEM DO BANCO (Oracle)

Use exatamente os nomes de tabelas e colunas abaixo:

**job_concurrency_control** — controle de execução de jobs
- idt_job_schedules, des_job_name, des_status (PENDING|RUNNING|COMPLETED), flg_active, dat_last_execution, idt_job_execution, idt_datacenter

**server** — servidores de origem/destino (S3, SFTP, NFS, Blob-Storage, Object Storage)
- idt_server, cod_server, cod_vault, des_vault_secret, des_server_type, des_server_origin (INTERNO|EXTERNO), flg_active

**sever_paths** — diretórios dentro dos servidores
- idt_sever_path, idt_server, idt_acquirer, des_path, des_path_type (ORIGIN|DESTINATION), flg_active

**sever_paths_in_out** — mapeamento origem → destino
- idt_sever_paths_in_out, idt_sever_path_origin, idt_sever_destination, des_link_type (PRICIPAL|SECUNDARIO), flg_active

**layout** — layouts de arquivo por adquirente
- idt_layout, cod_layout, idt_acquirer, des_version, des_file_type (CSV|TXT|JSON|OFX|XML), des_transaction_type (FINANCEIRO|CAPTURA|COMPLETO|AUXILIAR), des_distribution_type, flg_active

**layout_identification_rule** — regras para identificar o layout
- idt_rule, idt_layout, des_value_origin (HEADER|TAG|FILENAME|KEY), des_criterion_type_enum (COMECA-COM|TERMINA-COM|CONTEM|IGUAL), num_starting_position, num_ending_position, des_value, des_tag, des_key, flg_active

**customer_identification** — vínculo cliente + adquirente + estabelecimento
- idt_customer_identification, idt_client, idt_acquirer, idt_merchant, dat_start, dat_end, flg_is_prioritary, num_processing_weight, flg_active

**customer_identification_rule** — regras para identificar o cliente pelo nome do arquivo
- idt_rule, idt_customer_identification, des_criterion_type_enum (COMECA-COM|TERMINA-COM|CONTEM|IGUAL), num_starting_position, num_ending_position, des_value, flg_active

**file_origin** — registro de cada arquivo coletado
- idt_file_origin, idt_acquirer, idt_layout, des_file_name, num_file_size, des_file_type, des_transaction_type, dat_timestamp_file, idt_sever_paths_in_out, flg_active
- Índice único: (des_file_name, idt_acquirer, dat_timestamp_file, flg_active)

**file_origin_client** — associação arquivo → cliente identificado
- idt_file_origin_client, idt_file_origin, idt_client, flg_active

**file_origin_client_processing** — rastreabilidade de cada etapa do processamento
- idt_file_origin_processing, idt_file_origin_client, des_step (COLETA|RAW|STAGING|ORDINATION|PROCESSING|PROCESSED|DELETE), des_status (EM_ESPERA|PROCESSAMENTO|CONCLUIDO|ERRO), des_message_error, des_message_alert, dat_step_start, dat_step_end, jsn_additional_info, flg_active

## REGRAS DE NEGÓCIO CRÍTICAS

1. Identificação de cliente: aplicar todas as regras ativas de customer_identification_rule para o customer_identification candidato. Só identifica se TODAS retornarem true.

2. Identificação de layout: aplicar todas as regras ativas de layout_identification_rule. Origem pode ser FILENAME (nome do arquivo) ou HEADER (primeiros 7000 bytes lidos via streaming).

3. Upload com streaming: nunca carregar o arquivo inteiro em memória. Usar InputStream encadeado do SFTP diretamente para o destino (S3 multipart upload ou SFTP output stream).

4. Rastreabilidade: toda mudança de step/status deve gerar registro em file_origin_client_processing com dat_step_start, dat_step_end e jsn_additional_info quando relevante.

5. Credenciais de servidores (SFTP, S3) devem ser lidas do Vault usando cod_vault + des_vault_secret da tabela server. Nunca hardcodar credenciais.

## O QUE PRECISO QUE VOCÊ IMPLEMENTE

Implemente de forma incremental, começando por:

1. Estrutura do projeto (mono repo, módulos Maven ou Gradle)
2. Docker Compose com: Oracle XE, RabbitMQ, LocalStack (S3), SFTP server
3. Scripts DDL Oracle para todas as tabelas acima
4. Pod Orquestrador: scheduler + listagem SFTP + registro no banco + publicação RabbitMQ
5. Pod Processador: consumer RabbitMQ + download streaming + identificação cliente/layout + upload streaming + atualização de rastreabilidade
6. Manifests Kubernetes (Deployments, Services, ConfigMaps, Secrets)

A cada etapa, aplique: tratamento de erros, logs estruturados, testes unitários para as regras de identificação, e boas práticas Spring Boot (profiles, externalized config, health checks).

Pergunte se tiver dúvidas antes de implementar. Prefira código limpo e simples a soluções over-engineered.
