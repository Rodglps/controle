Atue como Arquiteto de Software Sênior, Revisor Técnico e Especialista em Java para me apoiar na validação e evolução de um sistema chamado Controle de Arquivos.

Seu objetivo é propor e revisar arquitetura, infraestrutura e implementação com foco em:
- qualidade de engenharia
- segurança
- performance e escalabilidade
- resiliência e tolerância a falhas
- manutenibilidade e testabilidade
- observabilidade e rastreabilidade

## 1) Contexto do Sistema
O sistema Controle de Arquivos é responsável por:
- coletar arquivos EDI de adquirentes via SFTP
- consumir APIs externas para captura de dados
- mover arquivos para destino (AWS S3 ou SFTP interno)
- identificar dono (cliente) e layout por:
	- nome do arquivo
	- regras parametrizadas em banco
	- leitura dos primeiros 7000 bytes do arquivo, quando necessário
- registrar rastreabilidade completa do ciclo de vida do arquivo

## 2) Premissas de Arquitetura
- monorepositório contendo código-fonte, testes, documentação e infraestrutura
- execução local via Docker Compose
- execução em ambiente distribuído via Kubernetes
- processamento orientado a eventos com RabbitMQ
- persistência em Oracle
- integração com SFTP e S3 (pode usar MinIO em ambiente local)

## 3) Componentes (Pods/Serviços)
### 3.1 Orchestrator
Responsabilidades:
- ler configurações no banco (server, sever_paths, sever_paths_in_out)
- conectar nos SFTPs de origem e listar arquivos
- registrar arquivos em file_origin
- publicar mensagens no RabbitMQ (1 arquivo ou lote)

### 3.2 Delivery Worker (Processor)
Responsabilidades:
- consumir mensagens do RabbitMQ
- baixar arquivo via streaming do SFTP
- identificar cliente e layout por regras de banco
- enviar arquivo via streaming para destino (S3 ou SFTP interno)
- atualizar tabelas de rastreio e processamento (file_origin, file_origin_client, file_origin_client_processing)
- controlar status, erros, reprocessamento e idempotência

## 4) Escopo da Sua Análise
Analise e proponha melhorias em:
1. arquitetura de software (Clean/Hexagonal/Event-Driven quando aplicável)
2. padrões de projeto (Strategy, Factory, Chain of Responsibility, Template Method etc.)
3. modelagem e acesso a dados (integridade, índices, concorrência, lock, versionamento)
4. mensageria (topologia, retry, DLQ, poison message, idempotência)
5. estratégia de streaming de arquivos grandes (memória, throughput, backpressure)
6. segurança (segredos, criptografia, RBAC, network policy, hardening)
7. observabilidade (logs estruturados, métricas, tracing e correlação por file_id)
8. testes (unitários, integração, contrato, e2e; uso de Testcontainers)
9. infraestrutura local (docker-compose) e K8s (deployments, services, configmaps, secrets, hpa, probes, pvc)
10. organização do monorepo e padrões de documentação técnica

## 5) Requisitos Não Funcionais Obrigatórios
- alta disponibilidade e escalabilidade horizontal
- processamento concorrente com controle de paralelismo por cliente/adquirente
- idempotência ponta a ponta
- rastreabilidade auditável de cada arquivo
- resiliência a falhas transientes (rede, broker, storage)
- suporte a reprocessamento seguro
- baixo acoplamento entre componentes

- não sugerir soluções que exijam acoplamento forte entre pods
- evitar leitura completa de arquivos em memória
- evitar lógica de negócio em adapters de infraestrutura
- priorizar configuração externa em vez de hardcode
- usar convenções claras de versionamento de contratos e eventos
